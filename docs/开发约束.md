# FastAPI 异步缓存工具开发约束

本文件用于约束后续 `kmcache` 项目的开发行为，确保代码在以下方面保持一致：

- 符合 Python 主流工程规范
- 满足异步缓存组件的可维护性要求
- 适合后续打包为 `pip` 包
- 与 [技术方案以及架构.md](/c:/Users/Administrator/Desktop/project/python/kmcache/docs/技术方案以及架构.md) 保持一致

后续所有代码实现、重构、测试、示例、文档补充，均必须同时遵循：

- [技术方案以及架构.md](/c:/Users/Administrator/Desktop/project/python/kmcache/docs/技术方案以及架构.md)
- [开发约束.md](/c:/Users/Administrator/Desktop/project/python/kmcache/docs/开发约束.md)

若两者发生冲突，处理优先级如下：

1. `技术方案以及架构.md` 决定系统设计、模块边界、能力范围
2. `开发约束.md` 决定编码风格、工程规范、实现细节约束

## 2. Python 版本与通用标准

### 2.1 Python 版本

- 目标 Python 版本为 `3.14.5`
- 编码实现应优先采用 Python 新版稳定语法与标准库能力
- 不为过旧 Python 版本做兼容性妥协

### 2.2 基础规范

后续实现必须尽量符合以下标准：

- `PEP 8`：代码风格规范
- `PEP 257`：文档字符串规范
- `PEP 484`：类型标注规范
- `PEP 544`：协议 `Protocol` 设计规范
- `PEP 561`：类型信息分发兼容性意识

### 2.3 代码总体原则

- 优先清晰性，而不是炫技式写法
- 优先可维护性，而不是过度抽象
- 优先组合，而不是深层继承
- 优先显式，而不是隐式魔法
- 优先小而稳定的公共 API，而不是一次性暴露过多能力

## 3. 工程设计约束

### 3.1 必须遵循架构文档

代码实现必须围绕 `技术方案以及架构.md` 中的以下约束展开：

- `L1`、`L2` 后端解耦
- 通过 `CacheManager` 完成统一编排
- 广播、熔断、预热、序列化等能力横向拆分
- FastAPI 集成为独立模块
- 后续支持包化与可选依赖

禁止出现以下实现方式：

- 将 L1/L2 强耦合写死在单一后端类中
- 业务逻辑直接散落到缓存底层实现中
- 用继承链硬编码所有模式组合
- 为了“方便”而绕过抽象接口直接跨层调用内部状态

### 3.2 模块职责单一

每个模块必须保持职责边界清晰：

- `backends/` 只负责缓存存取语义
- `features/` 只负责横切能力
- `integrations/` 只负责框架集成
- `observability/` 只负责指标、日志、追踪
- `utils/` 只放稳定、通用、无业务语义的辅助函数

禁止把以下内容混在一起：

- Redis 连接管理和业务回源逻辑
- 本地缓存结构和 FastAPI 路由依赖逻辑
- 熔断器状态机和序列化实现

## 4. 命名规范

### 4.1 通用命名

- 变量、函数、模块文件名使用 `snake_case`
- 类名使用 `PascalCase`
- 常量使用 `UPPER_SNAKE_CASE`
- 私有属性或内部函数使用前缀 `_`

示例：

```python
DEFAULT_TTL = 300


class LocalCacheBackend:
    async def get_or_load(self, key: str) -> object | None:
        ...


def build_cache_key(namespace: str, raw_key: str) -> str:
    ...
```

### 4.2 命名语义要求

命名必须表达真实职责，避免含糊缩写。

推荐：

- `local_backend`
- `redis_backend`
- `cache_manager`
- `soft_expire_at`
- `hard_expire_at`
- `singleflight_group`

避免：

- `obj`
- `tmp`
- `data2`
- `mgr`
- `do_cache`
- `handle_it`

### 4.3 布尔命名

布尔值字段或变量应使用明确语义：

- `is_expired`
- `is_stale`
- `has_value`
- `enable_broadcast`
- `use_distributed_lock`

避免：

- `flag`
- `status`
- `switch`

## 5. 注释与文档字符串规范

### 5.1 注释原则

注释只解释“为什么”或“不直观的实现约束”，不解释显而易见的代码动作。

推荐：

```python
# Use soft TTL to return stale data while a background refresh is in progress.
```

避免：

```python
# Assign key to variable.
key = item.key
```

### 5.2 注释语言

统一要求：

- 代码内注释、文档字符串优先使用英文
- `docs/` 下的方案和规范文档可以使用中文

原因：

- 代码未来可能打包为公共库
- 英文注释更利于开源和协作

### 5.3 文档字符串要求

以下对象必须写文档字符串：

- 公共类
- 公共函数
- 公共异步方法
- 装饰器
- 对外暴露的数据结构

文档字符串要求：

- 使用三引号
- 简洁说明职责
- 必要时补充参数、返回值、异常说明
- 不写无意义模板化废话

推荐：

```python
async def get_or_load(key: str, loader: Loader) -> object:
    """Return cached data or load it once on cache miss."""
```

## 6. 类型标注规范

### 6.1 强制类型标注

后续项目代码必须尽量完整标注类型，尤其是：

- 公共函数
- 公共类方法
- 协议接口
- 配置对象
- 复杂返回值

### 6.2 类型风格

优先使用现代 Python 类型语法：

- `str | None`，不使用 `Optional[str]`
- `list[str]`，不使用 `List[str]`
- `dict[str, int]`，不使用 `Dict[str, int]`

仅在兼容性或复杂泛型表达确有必要时再引入 `typing` 旧风格别名。

### 6.3 Any 使用约束

避免泛滥使用 `Any`。

允许场景：

- 后端协议的通用值类型
- 序列化边界
- 外部依赖类型不足时的临时兼容层

如果用了 `Any`，应确保理由充分，而不是为了省事跳过建模。

## 7. 异步开发规范

### 7.1 异步优先

公共 API 统一使用异步接口：

- `async def get(...)`
- `async def set(...)`
- `async def delete(...)`
- `async def warmup(...)`

即使底层某些操作是纯内存，也保持异步接口一致性。

### 7.2 禁止阻塞事件循环

禁止在异步链路中直接做以下行为：

- 长时间同步 CPU 密集计算
- 同步网络 I/O
- 同步 Redis 客户端调用
- 长时间持有全局锁

### 7.3 锁使用规范

- 只在必要临界区加锁
- 优先细粒度锁，而不是粗粒度全局锁
- 明确区分单 key 锁与全局锁
- 避免死锁和锁顺序不一致

### 7.4 后台任务约束

如果实现后台刷新、广播订阅、预热任务：

- 必须可启动
- 必须可关闭
- 必须处理取消信号
- 必须避免静默吞掉异常

## 8. 异常处理规范

### 8.1 不允许裸 `except`

禁止：

```python
try:
    ...
except:
    ...
```

必须捕获明确异常，并区分：

- 可恢复异常
- 需要熔断的异常
- 需要上抛的异常

### 8.2 自定义异常

缓存库内部应定义清晰异常层级，例如：

- `CacheError`
- `BackendError`
- `SerializationError`
- `CircuitBreakerOpenError`
- `LockAcquisitionError`

禁止直接把底层实现异常原样扩散到所有公共 API。

### 8.3 异常信息要求

异常消息必须具备可排障性，应包含必要上下文，例如：

- backend 名称
- key 或 key 模式
- 操作类型
- 原始异常摘要

但要避免把敏感业务数据直接写入异常或日志。

## 9. 日志规范

### 9.1 日志原则

- 日志服务于排障，不制造噪音
- 关键路径要可观测
- 高频缓存命中路径不要默认打印冗余 info 日志

### 9.2 建议日志级别

- `debug`：命中细节、回填细节、调试信息
- `info`：组件启动、关闭、预热完成、广播订阅建立
- `warning`：降级、返回 stale、锁获取失败、广播失败
- `error`：后端错误、序列化错误、关键任务失败

### 9.3 日志字段

建议统一上下文字段：

- `backend`
- `key`
- `event`
- `latency_ms`
- `instance_id`
- `trace_id`

## 10. 配置规范

### 10.1 配置对象化

禁止大量散落的参数传递。

优先使用：

- `dataclass`
- 或 `pydantic` 模型

配置要按领域拆分，例如：

- `CacheConfig`
- `LocalCacheConfig`
- `RedisCacheConfig`
- `CircuitBreakerConfig`
- `BroadcastConfig`

### 10.2 默认值要求

所有公共配置项必须有合理默认值，默认值应满足：

- 本地开发可直接运行
- 生产环境不会明显危险
- 没有隐藏的高风险行为

例如：

- TTL 不能默认无限大
- 本地缓存容量不能默认无限制
- 广播功能不能默认强制开启

## 11. 序列化规范

### 11.1 序列化边界明确

缓存值的编码/解码必须由独立序列化层处理，禁止在各后端中重复散写。

### 11.2 默认策略

第一版建议默认：

- 优先 `json` 兼容类型
- 保留可扩展接口支持 `msgpack` 等方案

### 11.3 安全性约束

若未来支持 `pickle`：

- 必须明确标识其不适合不可信输入
- 不得作为默认序列化方案

## 12. 测试规范

### 12.1 必须测试

后续新增功能时，原则上应同步新增测试，至少覆盖：

- 正常路径
- 边界条件
- 失败路径
- 并发行为

### 12.2 测试命名

测试函数名必须表达行为，例如：

- `test_local_backend_returns_none_for_missing_key`
- `test_cache_manager_backfills_l1_after_l2_hit`
- `test_get_or_load_uses_singleflight_for_hot_key`

### 12.3 测试隔离

- 单元测试不能依赖外部真实服务
- Redis 集成测试应显式标记
- 并发测试要尽量减少随机性

## 13. 代码组织规范

### 13.1 文件大小控制

单文件应尽量控制复杂度。

建议：

- 超过约 `300~500` 行时，评估是否应拆分
- 若一个模块承担多个职责，应优先拆模块，而不是继续堆代码

### 13.2 公共 API 控制

导出的公共 API 必须克制。

原则：

- 只暴露稳定、必要的接口
- 内部实现细节不要轻易公开
- `__init__.py` 不应无节制导出所有对象

### 13.3 向后兼容意识

既然目标是未来打包成公共包，则后续每个公共 API 都应具备版本演进意识：

- 参数命名要稳定
- 默认行为要保守
- 不轻易修改公开语义

## 14. 性能与可维护性约束

### 14.1 先正确，再优化

第一版实现优先保证：

- 语义正确
- 并发安全
- 降级清晰

在此基础上再做性能优化。

### 14.2 禁止过早优化

禁止在没有数据支撑前引入：

- 过度复杂的微优化
- 难以理解的无锁技巧
- 不必要的元编程

### 14.3 热路径意识

虽然不鼓励过度优化，但缓存组件属于高频路径，后续实现时仍需注意：

- 减少重复序列化
- 减少不必要对象复制
- 避免命中路径重日志
- 避免读路径被全局锁串行化

## 15. 第三方依赖约束

### 15.1 依赖选择原则

- 优先成熟、活跃、长期维护的依赖
- 优先标准库
- 避免引入重量级但收益有限的包

### 15.2 Redis 依赖约束

实际 Redis 异步实现建议基于：

- `redis.asyncio`

原因：

- 更符合当前生态
- 更适合作为公共包基础设施
- 比独立 `aioredis` 更稳

### 15.3 可选依赖

如未来打包为 `pip` 包，应将可选能力拆为 extras，而不是强制所有用户安装全部依赖。

## 16. 提交实现时的执行要求

后续每次新增代码时，都必须检查以下问题：

1. 是否符合 `技术方案以及架构.md` 的模块分层与职责设计？
2. 是否符合本文件的命名、注释、类型、异步、异常规范？
3. 是否引入了不必要耦合？
4. 是否为公共 API 提供了合理类型和文档字符串？
5. 是否考虑了测试与后续包化？

本项目后续开发必须坚持以下底线：

- 设计上以 `技术方案以及架构.md` 为准
- 编码上以 `开发约束.md` 为准
- 所有实现优先服务于可维护、可测试、可扩展、可包化

从下一步开始，后续代码编写均默认依赖并遵循这两份文档，不再单独重复声明。
