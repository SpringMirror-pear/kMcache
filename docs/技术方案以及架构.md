# FastAPI 异步缓存工具技术方案以及架构

## 1. 目标

设计并实现一个面向 FastAPI 的异步缓存工具，满足以下目标：

- 支持 `L1` 本地缓存模式。
- 支持 `L2` Redis(`aioredis`/`redis.asyncio`) 缓存模式。
- 支持 `L1 + L2` 两级缓存模式。
- 三种模式在架构上解耦，可按需独立启用。
- 支持生产环境关键能力：
  - 缓存雪崩防护
  - 缓存击穿防护
  - 缓存预热
  - 缓存广播通知（可选）
  - 熔断器
  - 异步接口
  - 面向 FastAPI 的易集成能力
- 具备后续打包为 `pip` 包的清晰演进路径。
- 兼容 Python `3.14.5`。

## 2. 设计原则

### 2.1 分层解耦

缓存能力拆分为以下层次：

- `CacheBackend`：单一缓存后端抽象，不关心业务，只负责 `get/set/delete/...`
- `CacheManager`：组合多个后端，负责 L1/L2 协同、回源、并发控制、降级
- `Feature Components`：广播、熔断、预热、序列化、统计等横切能力
- `FastAPI Integration`：依赖注入、生命周期管理、装饰器、路由级使用方式

核心原则是：**后端能力与编排能力分离**。  
也就是说：

- 只用 L1 时，不需要 Redis 依赖
- 只用 L2 时，不需要本地缓存实现参与读写路径
- 用 L1+L2 时，通过组合编排，不通过继承耦合

### 2.2 面向接口编程

采用协议/抽象基类定义能力边界：

- `BaseCacheBackend`
- `BaseSerializer`
- `BaseBroadcaster`
- `BaseCircuitBreaker`
- `BaseMetricsHook`
- `BaseWarmupProvider`

这样后续替换 Redis、接入别的本地缓存实现、接入消息中间件时，不影响上层调用。

### 2.3 异步优先

整个公共 API 统一使用 `async def`。  
即使 L1 本地缓存是内存操作，也保持异步接口，原因如下：

- 对 FastAPI 使用者心智统一
- 便于未来在链路中增加锁、限流、统计、广播等异步动作
- 避免同步/异步两套 API 带来的维护成本

## 3. 目标使用方式

### 3.1 仅 L1

```python
cache = CacheManager(
    backends=[
        LocalCacheBackend(...)
    ]
)
```

### 3.2 仅 L2

```python
cache = CacheManager(
    backends=[
        RedisCacheBackend(...)
    ]
)
```

### 3.3 L1 + L2

```python
cache = CacheManager(
    backends=[
        LocalCacheBackend(...),
        RedisCacheBackend(...),
    ]
)
```

这里约定：

- 第一个后端视为更快层，优先读取
- 后续后端视为更慢层，逐层回退
- 命中下层后，自动回填上层

## 5. 核心接口设计

## 5.1 缓存后端抽象

```python
from typing import Protocol, Any


class BaseCacheBackend(Protocol):
    name: str

    async def get(self, key: str) -> Any | None: ...
    async def set(
        self,
        key: str,
        value: Any,
        ttl: int | None = None,
    ) -> None: ...
    async def delete(self, key: str) -> None: ...
    async def exists(self, key: str) -> bool: ...
    async def expire(self, key: str, ttl: int) -> None: ...
    async def clear(self) -> None: ...
    async def close(self) -> None: ...
```

可选增强接口：

```python
class SupportsBatch(Protocol):
    async def mget(self, keys: list[str]) -> list[Any | None]: ...
    async def mset(self, mapping: dict[str, Any], ttl: int | None = None) -> None: ...


class SupportsLock(Protocol):
    async def acquire_lock(self, key: str, timeout: float) -> bool: ...
    async def release_lock(self, key: str) -> None: ...
```

说明：

- 后端只做存储语义，不承担复杂策略
- 击穿保护、预热、广播、熔断都由上层 `CacheManager` 或 `features` 处理

## 5.2 CacheManager 核心职责

`CacheManager` 负责：

- 多级缓存读写编排
- miss 回源
- 单飞控制（single-flight）
- TTL 抖动
- 失败降级
- 熔断控制
- 广播失效通知
- 预热任务调度
- 统一指标打点

建议公共接口：

```python
class CacheManager:
    async def get(self, key: str, default: Any = None) -> Any: ...
    async def set(self, key: str, value: Any, ttl: int | None = None) -> None: ...
    async def delete(self, key: str) -> None: ...
    async def get_or_load(
        self,
        key: str,
        loader,
        ttl: int | None = None,
        soft_ttl: int | None = None,
    ) -> Any: ...
    async def warmup(self, items: list["WarmupItem"]) -> None: ...
    async def close(self) -> None: ...
```

其中 `get_or_load` 是最重要的生产接口，用于解决大多数业务场景：

- 先查缓存
- miss 后触发并发安全的回源
- 成功后回填缓存
- 必要时支持 stale 数据兜底

## 6. 三种部署模式的解耦设计

## 6.1 模式一：仅 L1

特点：

- 极低延迟
- 无外部依赖
- 适合单实例应用、配置缓存、热点小数据

限制：

- 多实例间数据不一致
- 进程重启后数据丢失
- 不适合强一致共享缓存

## 6.2 模式二：仅 L2

特点：

- 多实例共享缓存
- 数据一致性更容易维护
- 天然支持分布式锁、Pub/Sub

限制：

- 每次读都要走网络
- 热点场景延迟高于 L1
- Redis 异常会直接影响缓存层

## 6.3 模式三：L1 + L2

推荐作为生产默认模式。

读取流程：

1. 先查 L1
2. L1 miss 后查 L2
3. L2 命中则回填 L1
4. L2 miss 则触发回源加载
5. 回源成功后同时写入 L2，再按策略写入 L1

写入流程：

1. 优先写 L2 作为共享事实来源
2. 再写 L1
3. 如果启用广播，则发布失效/更新事件给其他实例

删除流程：

1. 删除 L2
2. 删除当前实例 L1
3. 广播其他实例清理对应 L1

## 7. 关键特性设计

## 7.1 缓存雪崩防护

雪崩是大量 key 在同一时间失效，导致请求同时打到后端。

解决策略：

### 7.1.1 TTL 抖动

为每个 key 的 TTL 增加随机偏移量：

```text
actual_ttl = base_ttl + random(0, jitter)
```

作用：

- 避免同一批缓存同时过期

### 7.1.2 分级 TTL

对不同类型 key 使用不同 TTL 策略：

- 热点数据：短 TTL + 预刷新
- 冷数据：长 TTL
- 配置数据：长 TTL + 主动广播失效

### 7.1.3 过期后兜底旧值

引入：

- `hard_ttl`：真正失效时间
- `soft_ttl`：建议刷新时间

当超过 `soft_ttl` 但未超过 `hard_ttl` 时：

- 返回旧值
- 异步触发刷新

这可以大幅降低瞬时雪崩风险。

## 7.2 缓存击穿防护

击穿是热点 key 失效瞬间，大量并发请求同时回源。

解决策略：

### 7.2.1 SingleFlight

同一个 key 的并发 miss，只允许一个协程执行 `loader`，其他协程等待结果。

适用范围：

- 单进程内击穿防护

### 7.2.2 分布式锁

在多实例部署下，仅靠进程内 single-flight 不够。  
若启用 Redis，则可为热点 key 加 Redis 分布式锁。

策略：

- 实例 A 获得锁，负责回源
- 其他实例短暂等待并轮询缓存
- 超时则降级返回 stale 或默认值

### 7.2.3 热点永不过期 + 后台刷新

对于极热点 key，可以采用：

- 缓存逻辑过期
- 值可读
- 后台异步刷新

避免用户请求直接触发后端抖动。

## 7.3 空值缓存

用于防止缓存穿透。

策略：

- loader 返回空结果时，缓存一个显式空标记
- 空标记使用较短 TTL
- 区分“未命中”和“命中空值”

需要引入内部哨兵对象：

```python
class NullValue:
    pass
```

## 7.4 缓存预热

预热能力分为三种：

### 7.4.1 启动预热

FastAPI 启动时，主动加载一批高频 key。

适合：

- 配置
- 字典表
- 首页热点数据

### 7.4.2 定时预热

通过后台任务周期性刷新热点 key。

适合：

- 榜单
- 统计聚合
- 首页推荐

### 7.4.3 被动预热

根据访问统计识别热点 key，自动加入预热名单。

后续可作为增强能力，不建议第一版就做复杂自学习。

建议抽象：

```python
class WarmupItem:
    key: str
    ttl: int
    loader: callable
    soft_ttl: int | None = None
```

## 7.5 广播通知

用于多实例 L1 一致性维护，可选择开启。

设计目标：

- 不启用广播时，系统仍然可用
- 启用后，实例间可同步失效本地缓存

实现方式：

### 7.5.1 Redis Pub/Sub

当执行：

- `set`
- `delete`
- `expire`
- `clear`

可发布事件：

```json
{
  "event": "delete",
  "key": "user:1001",
  "source": "instance-a",
  "ts": 1710000000
}
```

其他实例订阅该频道后：

- 跳过自己发送的事件
- 清理本地 L1

说明：

- 广播只处理 L1 失效同步
- 不建议通过广播直接传输大对象

## 7.6 熔断器

熔断器主要保护 Redis 或后端加载器故障时的系统稳定性。

需要保护的对象：

- `RedisCacheBackend`
- 广播组件
- 回源 `loader`

熔断器状态：

- `CLOSED`：正常
- `OPEN`：短路，不再访问故障组件
- `HALF_OPEN`：探测恢复

策略建议：

- 连续失败阈值：可配置，如 `5`
- 打开时长：可配置，如 `30s`
- 半开探测次数：可配置，如 `1~3`

行为建议：

- Redis 熔断后：
  - L1 模式仍可工作
  - L1+L2 模式退化为仅 L1
  - 只 L2 模式可选择直接回源并短期不写缓存

## 7.7 Stale-While-Revalidate

建议作为核心特性支持。

读取逻辑：

1. 数据未过 `soft_ttl`，直接返回
2. 数据过 `soft_ttl` 但未过 `hard_ttl`，返回旧值并后台刷新
3. 数据超过 `hard_ttl`，按 miss 处理并触发同步回源

优点：

- 降低尾延迟
- 降低热点回源风暴
- 提高可用性

## 8. 本地缓存 L1 设计

## 8.1 建议实现

L1 可先使用自研轻量实现，后续可适配 `cachetools`。

核心数据结构：

```python
{
    key: CacheEntry(
        value=...,
        expire_at=...,
        soft_expire_at=...,
        version=...,
    )
}
```

需要支持：

- TTL
- soft TTL
- 最大容量
- 淘汰策略
- 并发安全

## 8.2 淘汰策略

第一版建议支持：

- `LRU`
- `TTL`

默认策略：

- 超时优先清理
- 容量满时按 LRU 淘汰

## 8.3 并发模型

L1 在 FastAPI 单进程内会遇到协程并发，建议：

- 使用 `asyncio.Lock` 保护关键写路径
- 对单 key 锁与全局锁分离

避免：

- 每次读都上全局锁
- 大范围串行化导致吞吐下降

## 9. Redis L2 设计

## 9.1 Redis 客户端选择

Python 新生态建议优先使用：

- `redis.asyncio`

原因：

- `aioredis` 已被并入 `redis-py`
- 维护状态更稳定
- API 更适合长期打包维护

文档中可保留“兼容 aioredis 风格接口”的设计说明，但实际实现建议基于 `redis.asyncio`。

## 9.2 Redis 存储格式

建议 value 统一封装为结构化对象：

```json
{
  "data": "...",
  "meta": {
    "created_at": 1710000000,
    "soft_expire_at": 1710000300,
    "hard_expire_at": 1710000600,
    "version": 1
  }
}
```

这样可以支持：

- stale 策略
- 元数据扩展
- 版本控制

## 9.3 Redis Key 规范

建议格式：

```text
{namespace}:{app}:{resource}:{identifier}
```

例如：

```text
cache:user-service:user:1001
```

好处：

- 便于隔离不同应用
- 便于批量管理
- 便于监控与排障

## 10. FastAPI 集成方案

## 10.1 生命周期管理

通过 FastAPI lifespan 创建与关闭缓存管理器：

```python
from contextlib import asynccontextmanager


@asynccontextmanager
async def lifespan(app):
    cache = CacheManager(...)
    app.state.cache = cache
    await cache.start()
    yield
    await cache.close()
```

## 10.2 依赖注入

提供依赖函数：

```python
def get_cache(request: Request) -> CacheManager:
    return request.app.state.cache
```

业务 handler 中直接注入。

## 10.3 装饰器支持

建议提供：

```python
@cached(key_builder=..., ttl=60)
async def get_user(user_id: int):
    ...
```

装饰器职责：

- 自动生成 key
- 自动调用 `get_or_load`
- 可选支持标签、命名空间、空值缓存、soft TTL

## 11. 配置模型设计

建议使用 `pydantic` 或标准库 `dataclass` 定义配置。

```python
class CacheConfig:
    enabled: bool = True
    namespace: str = "default"
    default_ttl: int = 300
    ttl_jitter: int = 30
    enable_broadcast: bool = False
    enable_stale: bool = True
    enable_null_cache: bool = True
    max_local_size: int = 10000
```

拆分子配置：

- `LocalCacheConfig`
- `RedisCacheConfig`
- `CircuitBreakerConfig`
- `BroadcastConfig`
- `WarmupConfig`

这样更适合未来打包成 `pip` 包。

## 13. 异常与降级策略

## 13.1 L1 异常

本地缓存异常通常不应影响主流程：

- 记录日志
- 直接降级为无 L1

## 13.2 L2 异常

Redis 异常处理建议：

- 熔断保护
- 优先返回 L1 或 stale 数据
- 无可用缓存时再回源

## 13.3 Loader 异常

回源失败时：

- 如果存在未过 `hard_ttl` 的旧值，则优先返回旧值
- 否则抛出业务异常

这比“缓存 miss 后直接失败”更适合生产环境。

## 17. 风险与注意事项

### 17.1 `aioredis` 命名风险

虽然需求里提到 `aioredis`，但从长期维护看，建议实现基于 `redis.asyncio`，原因是：

- 生态主流已迁移
- 后续兼容 Python 新版本更稳
- 更适合做公共包

### 17.2 本地缓存一致性

只要启用 L1，本地缓存跨实例不可能天然强一致。  
广播只能降低不一致窗口，不能承诺严格一致。

因此文档中需要明确：

- 该组件追求高性能和最终一致
- 不适用于强事务一致性场景

### 17.3 预热不要与业务强耦合

预热应通过 `loader` 回调或 provider 抽象实现，不要把业务逻辑硬编码进缓存库内部。

## 18. 建议的下一步落地顺序

1. 先定义抽象接口和配置模型。
2. 实现 `LocalCacheBackend`。

## 19. 结论

该缓存工具应采用“后端解耦 + 编排统一 + 特性插件化”的架构：

- `L1`、`L2` 是独立后端
- `L1+L2` 通过 `CacheManager` 组合
- 生产能力通过特性组件横向接入

这样可以同时满足：

- 单机场景轻量可用
- 多实例场景可扩展
- 与 FastAPI 良好集成
- 后续可稳定演进为通用 `pip` 包
